## 1. 项目理解

### 1.1 项目概述

LexiBridge AI Translator 是一个基于 LLM 的 Chrome 浏览器翻译扩展（Manifest V3），旨在解决技术内容翻译场景中"专有名词被误译"的痛点。通过引入 AI 翻译链路替代传统统计翻译，支持术语表、上下文提示词和多模型配置，提供更精准的专业内容翻译体验。

核心特色：
- **四大翻译场景**：划词翻译、整页翻译、PDF 翻译、YouTube 字幕翻译
- **AI 驱动**：基于 OpenAI 兼容 API，支持多供应商切换
- **术语可控**：用户自定义术语表，确保专有名词翻译一致性
- **场景统一**：多场景共用同一翻译引擎与术语配置

### 1.2 核心目标

按优先级排序：
1. **翻译准确性**：通过术语表和 AI 提示词工程，确保技术专有名词不被误译
2. **多场景覆盖**：统一支持网页、PDF、视频字幕三大主要技术内容消费场景
3. **可配置性**：用户可按预算和质量需求切换不同 AI 服务商
4. **隐私优先**：所有敏感配置本地存储，不依赖云端账号系统
5. **性能优化**：响应时间 < 1200ms，不打断用户阅读流

### 1.3 目标用户

**主要用户群体**：
- 技术开发者：阅读英文技术文档、API 文档、GitHub 内容
- 产品经理/研究人员：阅读行业报告、技术白皮书
- 技术视频观众：观看 YouTube 英文技术教程、会议录像
- 学术研究者：阅读英文论文、技术文档（PDF）

**用户痛点**：
- 传统翻译工具对技术术语翻译不准确（如 "server" 被译为"服务员"而非"服务器"）
- 不同场景需要切换不同翻译工具，体验割裂
- 付费翻译服务成本高，且无法自定义专业术语库

### 2.2 非功能需求

#### 性能要求
- **响应时间**：短文本（< 100 词）划词翻译 < 1200ms
- **吞吐量**：整页翻译 > 20 段/分钟
- **资源占用**：内存 < 250MB，CPU 额外占用 < 15%
- **并发支持**：本地插件，无并发用户限制

#### 安全要求
- **数据加密**：API Key 使用 `chrome.storage.local` 加密存储
- **权限最小化**：仅申请必要权限（`activeTab`, `storage`, `scripting`）
- **隐私保护**：不收集用户数据，不依赖远程账号系统
- **合规性**：符合 Chrome Web Store Program Policies

#### 可用性要求
- **图形界面**：设置页需提供直观的配置界面
- **无障碍访问**：支持键盘导航、屏幕阅读器
- **响应式设计**：适配不同分辨率屏幕
- **国际化**：界面本身支持中英双语

#### 可维护性要求
- **测试覆盖率**：单元测试 > 70%
- **代码规范**：ESLint + Prettier + TypeScript strict
- **文档完善**：用户手册、配置项文档、API 文档

### 2.3 需求澄清问题

请用户确认以下问题：

#### 1. **术语表作用域**
- **当前理解**：术语表是全局配置，但可按域名覆盖
- **需要确认**：
  - 是否需要支持"多个术语表同时生效"并按优先级合并？
  - 术语表是否需要支持正则表达式匹配（如匹配 `/\w+Service/` 这类模式）？

#### 2. **AI 模型调用策略**
- **当前理解**：每次翻译都调用 AI API
- **需要确认**：
  - 是否需要实现"混合模式"：短句用传统翻译（免费/快速），长句/技术术语才用 AI？
  - API 调用失败时的降级策略是什么？（显示错误提示 OR 降级到免费翻译服务？）

#### 3. **YouTube 字幕边界**
- **当前理解**：仅处理页面可见字幕，不破解 DRM
- **需要确认**：
  - 是否需要支持"字幕缓存"功能？（避免视频回放时重复翻译）
  - 字幕翻译是否需要支持"批量预取"（用户点击按钮翻译整集字幕）？

#### 4. **PDF 翻译深度**
- **当前理解**：优先支持 Chrome 内置 PDF 查看器，对扫描版 PDF 不承诺 OCR
- **需要确认**：
  - 是否需要支持"整段翻译"（用户点击段落，而非仅选词）？
  - 对扫描版 PDF 是否需要集成第三方 OCR 服务（如 Tesseract.js）？

#### 5. **MVP 范围确认**
- **当前理解**：2~3 周完成划词 + YouTube 字幕 + API 配置
- **需要确认**：
  - MVP 是否包含"翻译记忆库"？（影响成本优化效果）
  - MVP 是否需要支持"整页翻译"的基础版本？（影响用户价值感知）

## 4. 技术方案建议

### 4.1 推荐技术栈

基于需求分析和约束条件，推荐以下技术栈：

| 层级 | 技术选择 | 理由 |
|------|----------|------|
| **构建工具** | Vite + CRXJS | Vite 提供快速 HMR，CRXJS 原生支持 Chrome Extension MV3 开发 |
| **语言** | TypeScript 5.x | 严格类型检查，减少运行时错误，提升开发效率 |
| **前端框架** | React 18.x（仅设置页） | 设置页需要复杂表单交互，React 生态成熟 |
| **状态管理** | Zustand / Jotai | 轻量级状态管理，避免 Redux 过度设计 |
| **UI 组件** | shadcn/ui / Radix UI | 无障碍访问友好，样式可定制 |
| **样式方案** | Tailwind CSS | 快速构建响应式 UI，支持暗色模式 |
| **HTTP 客户端** | Fetch API / wretch | 轻量级，避免 axios 依赖过重 |
| **存储** | chrome.storage.local / sync | 官方 API，无需额外依赖 |
| **测试框架** | Vitest + Testing Library | 与 Vite 深度集成，测试体验好 |
| **代码规范** | ESLint + Prettier + TypeScript strict | 行业标准，工具链成熟 |

│                                                               │
└─────────────────────────────────────────────────────────────┘
                         │

              ┌─────────────────┐
              │  External APIs  │
              │  - OpenAI       │
              │  - Anthropic    │
              │  - 其他兼容API  │
              └─────────────────┘

**核心设计原则**：
1. **隔离性**：Content Script 负责页面交互，Background 负责业务逻辑，Options 负责配置
2. **可测试性**：核心翻译逻辑在 Service Worker 中，可独立单元测试
3. **可扩展性**：新增翻译场景只需添加新的 Content Script，复用 Background 翻译引擎
4. **容错性**：API 调用失败时降级到缓存或提示用户重试

### 4.3 关键技术决策

| 决策项 | 选择 | 原因 |
|--------|------|------|
| **构建工具** | Vite + CRXJS | Vite HMR 速度快，CRXJS 原生支持 MV3，避免手动配置 manifest |
| **状态管理** | Zustand | 轻量（< 1KB），API 简单，适合设置页小规模状态 |
| **UI 组件** | Radix UI | 无障碍访问原生支持，样式完全可控，符合 Chrome Web Store 要求 |
| **样式方案** | Tailwind CSS | 开发速度快，支持暗色模式，体积可通过 tree-shaking 优化 |
| **HTTP 客户端** | 原生 Fetch API | 零依赖，满足 OpenAI API 调用需求，避免 axios 体积过大 |
| **存储方案** | chrome.storage.local | 官方 API，支持对象存储，自动同步到用户登录的 Chrome |
| **测试框架** | Vitest | 与 Vite 深度集成，性能优于 Jest，配置更简单 |
| **术语表存储** | chrome.storage.local | 支持快速查询，容量充足（50MB），无需引入 IndexedDB |
| **翻译缓存** | chrome.storage.local + LRU | 本地缓存降低 API 调用成本，LRU 策略控制容量 |
| **PDF 处理** | Chrome 内置 PDF Viewer | 避免引入 PDF.js 依赖，减少包体积，满足基本需求 |

### 6.3 技术难点

1. **YouTube 字幕实时翻译**（难度：高）
   - 需要监听字幕变化，实时调用 API
   - 延迟控制在 1.5s 内，需要优化 API 调用策略
   - 双语字幕 DOM 注入不影响原视频播放

2. **整页翻译 DOM 保留**（难度：高）
   - 智能分块：识别段落边界，避免截断句子
   - 结构保护：不破坏代码块、链接、行内公式
   - 性能优化：大页面分批翻译，不阻塞主线程

3. **术语表优先级控制**（难度：中）
   - 精确匹配 > 前缀匹配 > 正则匹配
   - 多术语表合并时的冲突解决
   - 术语边界检测（避免 "service" 误匹配 "serviceWorker"）

4. **AI API 成本优化**（难度：中）
   - 短文本合并请求，降低 API 调用次数
   - 翻译记忆库缓存策略
   - 用户提示词优化（减少 token 消耗）
